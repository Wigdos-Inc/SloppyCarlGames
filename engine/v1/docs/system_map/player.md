# player/ — Player Character

## Responsibility
Owns all player state, movement logic, model assembly, abilities, and animation. `Master.js` is the central orchestrator; `Model.js` assembles the player as an entity via the shared builder. Handlers call into `Master.js` to drive per-frame updates; physics calls the shared `builder/NewEntity.js` model-sync function directly to re-pose the player after physics resolution, the same as any other entity.

## Files
- `Master.js` — Player state orchestrator. Extends a builder-assembled entity into player state via `createDefaultPlayerState` (adds action/movement/ability/checkpoint fields on top of the built entity; widens `collision.simRadiusPadding` to 24 since the player broadphases a larger radius than the entity default). Exports `InitializePlayer`, `UpdatePlayer`, `UpdatePlayerModel`, `ResolvePlayerState`, `SetPlayerAction`, `TriggerPlayerDeath`, `TriggerPlayerRespawnSequence`, `RespawnPlayer`, `GetPlayerState`, `GetPlayerInput`, and `PlayerAPI` (the `ENGINE.Level.Player` public surface). `SetPlayerAction` is the sole write path for `playerState.action` — it sets `action`/`previousAction`, logs the transition, and conditionally fires `PLAYER_ACTION_CHANGE`; all transition sites (`ResolvePlayerState`, `TriggerPlayerDeath`, `RespawnPlayer`, jump/stun triggers elsewhere) go through it rather than writing `action` directly.
- `Movement.js` — Per-frame movement handler. Translates input flags into velocity and displacement deltas.
- `Model.js` — Player entity assembler. `synthesizeDefinition` builds a standard entity-definition object from a `characters.json` template plus level-supplied placement data (spawn position, scale, animations, custom events); `BuildPlayerModel` passes it to `builder/NewEntity.js`'s `BuildEntity` with `textureScale: null` (opts the player out of the shared entity-part geometry/face-texture cache, matching the model's historical inline build path) and a dedicated zero-origin `playerSurfaceMap` (spawn position is absolute, so the surface origin is `(0,0,0)`). `RefreshPlayerModel` calls the shared `UpdateEntityModelFromTransform` to re-pose and recompute `collision.aabb`/`physics`/`hurtbox`/`hitbox` each frame. `filterDeadDecals` drops image decals that failed to load by shallow-copying only the affected part, so the shared `characters.json` template is never mutated. Owns the character-template IIFE: `characters.json` bypasses `core/normalize.js`, so this file's top-level `instanceCharacterTemplates` IIFE is the template's single Unit-instancing site (parts' `dimensions`/`localPosition`/`localRotation`/`pivot`, decal transforms, and tube `primitiveOptions` node fields). Exports `CharacterData`, `BuildPlayerModel`, `RefreshPlayerModel`.
- `Abilities.js` — Player ability definitions (jump, boost, etc.) and application logic. Accepts ability handler callbacks from game code.
- `Animation.js` — Player-specific animation handler. Drives visual state transitions based on player movement and action state.
- `characters.json` — Character preset definitions, one per playable character (including `carl`). Each entry supplies a `model.parts` list (standard entity part shape, including `tube` parts) and a `collisionOverride` consumed the same way as an entity's. Raw numeric values are instanced into `Unit`/`UnitVector3` by `Model.js`'s IIFE at module load; downstream code receives pre-instanced values.

## Boundaries
**Called by:** `handlers/game/Level.js` (`InitializePlayer`, `UpdatePlayer`, `ResolvePlayerState`, `GetPlayerState`); `physics/Master.js` (`TriggerPlayerRespawnSequence` only — model sync goes through `builder/NewEntity.js`'s `UpdateEntityModelFromTransform`, not a player-specific function).  
**Calls into:** `builder/NewEntity.js` (`BuildEntity`, `UpdateEntityModelFromTransform` — the player is built and re-posed by the same code path as any other entity); `math/` (Utilities, Vector3); `core/` (meta, config, normalize).  
**Does not:** Manage its own update loop; drive rendering directly (model updates are consumed by `Render.js`); apply physics forces (those are applied by `physics/Master.js`); build its own geometry/collision math (delegated to `builder/NewEntity.js`).

## Invariants
- There is one player state object in the engine at a time, held in `Master.js`.
- `playerInputFlags` is written by game code each frame via `ENGINE.Level.Player.Input` and read by the player update each frame.
- `characters.json` values are instanced once at module load by the top-level IIFE in `Model.js`. No downstream code re-instances them.
- The player carries no bespoke collision system — `collision.aabb`/`physics`/`hurtbox`/`hitbox` are the same builder-computed structure every entity has, recomputed each frame by `UpdateEntityModelFromTransform`. There is no separate `collision.profile` cache.
- `playerState.action` is written only through `SetPlayerAction`; no other module assigns `playerState.action` directly.
