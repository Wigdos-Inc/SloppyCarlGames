# builder/ — Scene and Asset Construction

## Responsibility
Constructs all visual and structural scene data: 3D WebGL meshes, entity models, obstacles, terrain, null-space void walls, procedural textures, scatter details, UI element trees, and the full level scene graph. Builders are called by handlers (and in one case by physics), produce output objects, and return. They hold no persistent state and drive no update loops.

## Files
- `NewLevel.js` — Level scene graph assembler. `BuildLevel` is the primary entry point. Orchestrates all other builders to produce the complete `sceneGraph` object consumed by the renderer and physics pipeline. Also exports `RefreshSceneBoundingBoxes`.
- `NewObject.js` — Primitive 3D mesh builder. Generates WebGL geometry (positions, normals, UVs, indices) for mesh primitives. Foundation for all other geometry builders. Entity-part builds (role `"entity-part"`, `textureScale !== null`) are memoized: `buildEntityPartGeometryTemplate` constructs and freezes an immutable `{ positions, indices, uvs, bounds, faceTextureGroups?, triplanar? }` template once per `blueprintId::partId` key, stored in a caller-supplied `geometryCache` Map and shared by reference across all same-blueprint instances; per-instance state (transform, material, worldAabb, detailedBounds, scatter, customTextures) is always built fresh. `textureScale: null` signals the player-model inline build path, which bypasses caching entirely. Noise-pattern entity parts set `template.triplanar = true` and skip both `faceTextureGroups` construction and UV frequency-scaling (UVs are left at their default values — unused by the triplanar shader but kept valid so the shared VAO uv-buffer layout is unchanged); `BuildObject` sets `material.textureScale` per-mesh for triplanar parts so density stays correct across geometry shared by same-blueprint instances. Sphere UVs for non-triplanar entity-parts are pre-scaled by physical span (U → π·dimensions.x, V → π·dimensions.y·0.5) before frequency UV scaling so textures read at a consistent world scale regardless of part radius. `buildSphere`/`buildCylinder`/`buildCapsule`'s surface parameterizations are mirrored by `Render.js`'s decal `projectToSurface` GLSL for curved-surface decal conforming — the two must stay in sync if either changes. The `tube` shape is a **node-chain bone walk** rather than a single extrusion: `resolveTubeNodes` starts at an identity root frame (carrying the part's own `size`/`thickness`/`curved`/`smoothness`) and composes each `primitiveOptions.nodes[]` entry onto the running frame, so a node's `localPosition` is applied in the previous node's rotated axes. `makeTubeNode` extracts each node's `center` (frame translation) plus its `forward` (frame Y axis), `xAxis` and `zAxis`. `buildTube` lays one hollow ring cross-section at every node (`appendTubeRing`), then walks consecutive pairs: if `nodeA.curved`, it samples a connector centerline via `SampleConnectorCenterline` and lofts intermediate rings along `ParallelTransportFrames`, linearly interpolating `radiusX`/`radiusZ`/`thickness` across the span; `orientConnectorFrames` then distributes a residual roll so the final frame's normal lands on `nodeB`'s authored `xAxis`, giving a seamless join. Rings are stitched pairwise (`stitchTubeRings`), the two chain ends are closed (`capTubeRing`), and per-connector face groups are emitted as angular sectors with radial normals (`appendTubeConnectorFaceGroups`), mirroring `buildTorus` rather than the outer/inner/top/bottom split that has no meaning for a multi-node tube. Exports `GenerateUVs` and `TransformPointByMatrix` (used by `NewVoid.js`).
- `NewEntity.js` — Generic animated entity builder. `BuildEntity(definition, surfaceMap, textureScale, faceTextureStore, geometryCache)` constructs a multi-part entity model with collision profile and returns `{ entity }`. Face textures generated during the build write directly into the caller-supplied `faceTextureStore` (no separate return). Exports `UpdateEntityModelFromTransform` for post-physics model synchronization.
- `NewObstacle.js` — Obstacle mesh builder. `BuildObstacles(source, options)` constructs obstacle geometry for scene placement; `options` carries `{ textureScale, faceTextureStore }` so per-face generated textures write into the same content-addressed store as terrain. Each obstacle record carries a `worldAabb`.
- `NewTexture.js` — Procedural 2D texture generator and visual resource manager. Five patterns: `noise`, `tiles`, `stripes`, `grid`, `radial`. Three element shapes for noise/tiles/grid cells: `circle`, `diamond`, square (default). **Frequency patterns** (`tiles`, `stripes`, `grid`) carry their spatial frequency in per-mesh UV tiling and do not bake `density` into the shared canvas — only `speckSize` is applied. **Non-frequency patterns** (`noise`, `radial`) bake `density` into the canvas directly. Per-face **face textures** (noise only, non-square canvas sized to face UV span in pixels) are **content-addressed**: `buildFaceTextureSignature` produces a deterministic key from the base texture id, resolved density/speckSize/shape/colors, face pixel size, and animation params; identical content shares one baked canvas and one GL texture. `BuildFaceTextureData(store, ...)` takes a content-addressed store as its first argument, dedups via the internal `getOrBuildFaceTexture` gate, and returns only `{ faceTextureGroups }`. `ResolveNoiseFaceBlueprint(textureBlueprint, textureDetail)` is the shared helper that composes density/speckSize, applies the detail shape, and applies conditional `primary`/`secondary` rgba color overrides for per-face noise baking — called by both the mesh per-face path and `NewVoid.js`. `ComputeGeneratedTextureID(generatedTexture)` derives a color-override-aware texture id (appending `p=` and `s=` rgba segments when overrides are present). `RgbaToHex` converts an rgba object to a hex string. Canvas-level `primary`/`secondary` rgba overrides replace template default colors at bake time; the shader tint for generated textures is neutral `{1,1,1,1}`. Each texture registry entry holds `{ id, definition, source (canvas), dirty }`. Entries with `definition.animation.able === true` are picked up by `handlers/game/Texture.js` for hold/blend animation; `definition.isFaceTexture` flags the rebuild path. `BuildNoiseAnimationOptions` derives animation config from a blueprint + textureDetail pair. `collectCustomTextures` walks a mesh/part's authored decal entries into a flat `customTextureUsage` list (carrying `opacity` for image entries) for registry construction. Image-type decals with `opacity < 1` are baked once at registry build time: `bakeImageDecalOpacity` composites the raw bitmap onto a straight-alpha 2D canvas via `globalAlpha`; `opacity >= 1` skips baking and uses the raw bitmap directly (fast path). `createTextureRegistry`'s custom-texture loop invokes this bake per image decal; shape decals and the render-time tint path are unaffected. `AddToVisualResources` incrementally registers textures for objects spawned mid-session. Exports `PrepareLevelVisualResources`, `AddToVisualResources`, `BuildTextureSurface`, `BuildNoiseFaceCanvas`, `BuildFaceTextureData`, `ResolveNoiseFaceBlueprint`, `BuildNoiseAnimationOptions`, `FREQUENCY_PATTERN_CONFIG`, `VISUAL_TEMPLATES`, `ComputeGeneratedTextureID`, `RgbaToHex`.
- `NewScatter.js` — Procedurally generated multi-part static scatter objects placed on terrain surfaces. Each object's open faces (from null-space relations, see `NewVoid.js`) are threaded through to `iterateScatterInstances`, which rejects any sample whose XZ footprint AABB overlaps a top-relevant open face's XZ AABB (conservative AABB-vs-AABB, reusing the footprint math shared with the parent-bounds check) so scatter is not generated over openings. Only root parts (`level === 0`) are footprint-checked.
- `NewUI.js` — HTML UI element builder (`UIElement`). Produces DOM element trees consumed by `Render.js`.
- `NewBoss.js` — Boss entity builder. Specializes entity construction for boss characters.
- `NewTerrain.js` — Terrain builder entry point. Currently an empty placeholder — exports no functional builders.
- `NewVoid.js` — Null-space void wall builder. `BuildVoidWalls(sceneGraph, textureScale, faceTextureStore)` partitions each null-space mesh's faces via `classifyFaces` (AABB overlap + narrowphase tests): embedded faces become renderable void walls, open/touching faces are retained as world-space `Triangle`s (`UnitVector3`, instanced at this boundary). Attaches a per-pair `relations` map onto each null-space entry in place and writes generated face textures directly into `faceTextureStore` — returns nothing. The void wall per-face noise path calls `ResolveNoiseFaceBlueprint` (same helper as mesh per-face), so void walls honor `primary`/`secondary` color overrides. Called post-build by `NewLevel.js`.
- `templates/` — JSON blueprint data: `characters.json`, `enemies.json`, `levels.json`, `obstacleBlueprints.json`, `projectiles.json`, `terrainBlueprints.json`, `textures.json`. Treated as part of the builder group; `core/normalize.js` may import from here for payload canonicalization (documented exception in MODULE_GROUPS.md).

## Boundaries
**Called by:** `handlers/` (Level.js, Cutscene.js, and game-layer handlers call builders to construct scene content); `physics/Master.js` imports `UpdateEntityModelFromTransform` from `NewEntity.js` for post-physics model sync.  
**Calls into:** `math/` (Matrix, Vector3, Utilities, Collision); `physics/Collision.js` (`NewVoid.js` uses `NarrowphaseTest`); `core/` (meta, config).  
**Does not:** Manage persistent state, drive update loops, or write to the DOM or WebGL directly — all output is consumed downstream by `Render.js`.

## Invariants
- Builders are constructors: given scene input data, they produce scene output objects. They do not hold their own persistent state. Callers supply `faceTextureStore` and `geometryCache` stores; builders write into them, but ownership stays with the caller.
- The `sceneGraph` produced by `NewLevel.js` is the authoritative scene representation shared by the renderer and the physics pipeline.
- `NewTerrain.js` is currently a placeholder and contributes no runtime behavior.
- Null-space data lives in a per-pair `relations` map on each null-space entry (terrain mesh or obstacle record): `relations[relatedObjectId] = { suppressed, openFaces: Triangle[], voidWallMeshes: VoidWallMesh[] }`. `relatedObjectId` is a terrain mesh id or an obstacle *record* id (never a part id); `relations` is always initialized (`{}` at minimum) so readers never guard the container. This replaces the removed `sceneGraph.voidWalls.{terrain,obstacles}` arrays and the removed `suppressedCandidateIds` sets (suppression is now `relations[id].suppressed`).
- Build order in `NewLevel.js` is load-bearing: terrain and obstacles are built first, then `BuildVoidWalls` runs (attaching `relations`, writing face textures into `faceTextureStore`), then an `openFaces` lookup (`Map<defaultObjectId, Triangle[]>`) is assembled from the relations, and only then are scatter batches enqueued. Scatter must run after void walls so it can reject samples over openings — it is no longer enqueued inline during terrain/obstacle construction.
- `faceTextureStore` is a content-addressed plain object (`{}`, keyed by face content signature) created in `NewLevel.js` and threaded through all builders. Builders write into it; `NewLevel.js` stores it as `sceneGraph.pendingFaceTextures`. At runtime spawn, `handlers/game/Level.js` passes the live `textureRegistry` directly as the store so new face textures dedup against already-resident entries without a separate merge step.
- `sceneGraph.partGeometryCache` is a level-scoped `Map` (keyed `blueprintId::partId`) that holds frozen immutable part geometry templates shared by reference across same-blueprint entity instances. It persists on the sceneGraph for the lifetime of the level so runtime spawns reuse already-computed geometry. `handlers/game/Simulator.js` resets it on each `Load` so edited blueprints always rebuild fresh geometry.
- Consumers read the `relations` model directly: `physics/Collision.js`, `handlers/Render.js`, and `handlers/game/Camera.js` iterate `relations → voidWallMeshes` (and `openFaces` for the stencil) instead of the removed flat arrays. The `Render.js` stencil pass also draws open faces (world-space tris, identity model matrix) with a negative `polygonOffset` to suppress coplanar z-fighting.
- A tube node's `localRotation` defines the **tangent at that node** — shared by the connector arriving into it and the connector departing it. It is not the heading of the next segment. A connector is well-conditioned when both of its node tangents sit symmetrically about its chord; when one tangent lies near the chord and the other far off it, the cubic must approach its endpoint from off-axis and the centerline visibly hooks. This is a property of the authored node data, not of the curve evaluator.
- A tube node owns only the connector that **departs** it: `buildTube` iterates `i < nodes.length - 1` and reads `nodes[i].curved` / `nodes[i].smoothness`. The final node departs no connector, so `curved` and `smoothness` authored on it are never read.
- `buildTube` reuses `resolveCylinderSegments(complexity)` (8 / 12 / 16) for two independent purposes: the radial resolution of each ring **and** the number of connector centerline samples. A tight connector is therefore resolved by at most 17 centerline points.
