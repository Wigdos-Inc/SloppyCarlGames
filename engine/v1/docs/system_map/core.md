# core/ — Engine Foundation

## Responsibility
The foundation layer. Provides logging, event dispatch, session persistence, global configuration, payload validation/normalization, and the public ENGINE API surface. Everything else in the engine depends on core/; core/ itself has minimal reverse dependencies. Two files participate in the startup chain outside the core/ directory: `Bootup.js` at the engine root (entry point) and `canonSchemas.json` embedded inside core/.

## Files
- `config.js` — Runtime configuration singleton. Physics constants (gravity, buoyancy, resistance), debug flags, volume levels, performance settings, camera FOV, custom event toggles. Imports `Unit` from `math/Utilities.js` to instance typed physical constants at load time.
- `ini.js` — Builds and returns the public `ENGINE` API object. Imports from handlers, player, and math and assembles them into namespaced groups. Called once by `Bootup.js`; the result is assigned to `globalThis.ENGINE`. Top-level shape: `Log`, `Config`, `Cache` (direct); `Meta` (log replay, event dispatch, session helpers, pointer-lock read/request, `CNU_SCALE`); `Input` (input router, pointer-lock read/request/release, `Cursor`); `Controls`; `Cutscene`; `Startup`; `UI`; `Audio`; `Level` (lifecycle functions + `Player` API); `Math` (sub-namespaced: `Convert`, `Vector3` — `AddVector3, DivideVector3, MultiplyVector3, ScaleVector3, DotVector3` —, `Instancing`, `Physics` — `ComputeGravity, ComputeResistance, ComputeBuoyancy, ComputeStepVelocity, ComputeSubmergence` —); `Simulator`; `Blueprints`. `Blueprints` is the authored (pre-instancing) shape of engine-owned template JSON — player characters, terrain/obstacle templates, character/enemy/projectile templates, scatter — for the external Simulator App; it is the `raw` clone returned by `builder/templates/Instance.js`'s `InstanceEngineTemplates()`, which `ini.js` calls directly (no longer imports the template JSON files itself).
- `meta.js` — Engine-wide utilities: `Log` (channel-aware, source-aware logging with session persistence and duplicate suppression), `SendEvent` (CustomEvent dispatch to `window`), session storage helpers (`PushToSession`/`ReadFromSession`), `Cache` (runtime state cache for UI, Level, Cutscene payloads), `Cursor`, `Wait`, `ExitGame`, `IsPointerLocked`, `RequestPointerLock`, `ReleasePointerLock`. Also exports constants `EPSILON`, `ENTITY_TYPES`, and the `SetEngineInitialized` lifecycle flag setter.
- `validate.js` — Validates incoming game payloads against `canonSchemas.json`. Returns structured error arrays without throwing. Imports `isTemplateRef` from `normalize.js` (single shared discriminator, not redefined locally) to branch object lists (`validateObjectList`), `entityBlueprints` bucket entries, and `ValidateSimulatorPayload`'s definition onto the `templateObjectRef`/`templateEntityRef` schemas instead of `levelObject`/`levelEntityBlueprint` when a raw entry's `shape` is `"template"`/`"blueprint"`. Template id existence is not checked here — that happens once downstream in `normalize.js`; `validate.js` only checks the ref's own shape is well-formed.
- `normalize.js` — Normalizes validated payload values: resolves aliases, applies fallbacks, instances `Unit`/`UnitVector3` from raw numbers. Runs a `hoistObjectColor` pre-pass before `normalizePayloadSchema` on `LevelPayload` (terrain objects, obstacles, entities, entity blueprint model parts) and `SimulatorPayload` (definition and definition.model.parts): if a node carries a root-level `color` field, it is moved into `texture.generated.primary` before schema processing, so game payloads may use a top-level `object.color` authored name while the internal `primary`/`secondary` contract is preserved; skipped for template references (`isTemplateRef(item)` true), whose `color` is a direct ref-level override, not a field to hoist. Imports character templates from `player/` and texture templates from `builder/templates/` for canonicalization lookups (approved exception in MODULE_GROUPS.md), plus (for the template-reference system below) all five `builder/templates/` template collections. Canonicalizes the otherwise-opaque `tube` `primitiveOptions` bag at this boundary so `builder/NewObject.js` needs no guards: `normalizeTubeOptions` instances the root `thickness` as a cnu `Unit`, coerces `curved` to a bool (fallback `false`) and clamps `smoothness` to `[0,1]` (fallback `0`), then maps every entry of `nodes[]` through `normalizeTubeNode`, which validates against the `tubeNode` schema and instances `dimensions`/`localPosition` as cnu `UnitVector3`, `localRotation` as degrees→radians, and `thickness` as a cnu `Unit`. Wired shape-conditionally (`shape === "tube"`) from both `normalizePart` and `normalizeLevelObject`. The Simulator blueprint-merge path re-hydrates the same tube fields after `structuredClone` strips the class prototypes, reading `localRotation` back as `"radians"` (already converted) rather than degrees.

  **Template reference normalization** (boundary lookups only — resolves ids and instances override fields, never constructs the full template definition; actual template resolution happens in `builder/NewTemplate.js` at build time). Exports `isTemplateRef(value)` (true when `value.shape` is `"template"` or `"blueprint"`) — the single discriminator used by both `validate.js` and `normalize.js` to branch payload entries onto the template-ref path instead of the ordinary object/blueprint path. `normalizeObjectTemplateRef(rawObject, ctx, collection, label)` validates the referenced `terrain`/`obstacle` template id exists (warn-and-drop otherwise), instances `position`/`rotation`/`scale`/`texture` override fields, and normalizes `parts` as **per-part overrides** (`normalizeTemplatePartOverrides`, keyed by part id, validated against the template's actual part id set — unknown ids warn-and-drop). `normalizeEntityTemplateRef(rawRef, ctx, label)` does the equivalent for `character`/`enemy`/`projectile` template refs living in an `entityBlueprints` bucket entry, instancing `model.rootTransform` with null-aware fields (an unset override field stays `null`, letting the template value win downstream) via the shared `normalizeRootTransform(transform, requireAll)` helper (also used by `normalizeBlueprint`/`normalizeOverride` with `requireAll: true`, where every field is always instanced). `mergeTemplateRefEntity` (called from `mergeBlueprintWithOverride` when `blueprint.shape === "template"`) merges an `entities`-list spawn override onto a template-ref blueprint — supports `id`/`blueprintId`/`hp`/`collisionOverride`/`dialogue`/`spawnSurfaceId`/`rootTransform` overrides; `movement`/`velocity`/`attacks`/`animations` overrides are not supported for template-ref spawns (template values always win). Both `LevelPayload`'s terrain/obstacle/entityBlueprints mapping and `SimulatorPayload` branch per-entry on `isTemplateRef` to call the ref normalizer instead of `normalizeLevelObject`/`normalizeBlueprint`; `SimulatorPayload` additionally warn-and-nulls a terrain template ref outright (the Simulator's single-mesh terrain spawn path cannot round-trip multi-part terrain).
- `canonSchemas.json` — Schema registry. Defines field types, allowed values, ranges, fallbacks, and aliases for all payload structures that pass through validate/normalize. Generated texture color is expressed as `primary` (aliases: `color`, `textureColor`; fallback: null) and `secondary` (alias: `color`; fallback: null) rgba override fields — these replace the template default colors at canvas-generation time rather than via shader tint. The `tubeNode` entry defines one node of a `tube` bone chain: `dimensions` (x/z cross-section opening radii; `y` is unused and present only for schema uniformity), `thickness`, `localPosition` (an offset expressed **in the previous node's rotated frame**), `localRotation`, `curved`, and `smoothness`. `templateObjectRef` and `templateEntityRef` are the schemas for `shape: "template"`/`"blueprint"` payload entries (terrain/obstacles, and entity-blueprint-bucket entries respectively): every overridable field (scale, color, texture, collisionShape, mode; rootTransform, hp, collisionOverride) falls back to `null` so the resolver can distinguish "not overridden" from an authored value, and both carry a `parts` array of per-part override entries (id + overridable fields only, no shape/geometry fields — composition and `repeat` are template-authored and have no payload schema representation at all).
- `engine/v1/Bootup.js` — Engine entry point. Exports `StartEngine()`. Runs browser context validation, calls `Initialize()` from `ini.js`, attaches `PlayIntroCinematic` to the ENGINE object, assigns ENGINE to `globalThis`, runs the startup overlay, splash sequence, and fires `UI_REQUEST` to signal the game.

## Boundaries
**Called by:** `Bootup.js` calls `ini.js`. Any engine module may import `meta.js` and `config.js`. Handlers call `validate.js` and `normalize.js` at payload entry points.  
**Calls into:** `math/Utilities.js` (`config.js`, `normalize.js` import `Unit`/`UnitVector3`); `math/Vector3.js` (`normalize.js` imports `CloneVector3`); `builder/templates/Instance.js` (`ini.js` calls `InstanceEngineTemplates()` for `Blueprints`). `ini.js` imports from handlers, player, and math to assemble the API surface — this is a one-time assembly import, not an ongoing dependency.  
**Does not:** Contain game-specific logic, rendering code, or physics calculations.

## Invariants
- `Log` is callable from any engine module at any time, including before full initialization.
- `Cache` is initialized from `sessionStorage` at module load and holds last-known UI, Level, and Cutscene payloads across same-tab navigation.
- `CONFIG` is a singleton. No module replaces or re-imports it with different values.
- `validate.js` never throws — it returns error arrays. Callers check results before proceeding.
- `normalize.js` produces `Unit`/`UnitVector3` instances. Downstream modules receive pre-instanced values and must not re-instance them (see UNIT_INSTANCING.md).
- `canonSchemas.json` is the single source of truth for allowed payload field names, types, and defaults.
- `globalThis.ENGINE` is set once by `Bootup.js` at startup and is the exclusive game-facing API surface.
